## The prompt

I'm building a VS Code extension called **Pip-Token** that helps Claude Code users track their token usage in real time. It's styled after the Pip-Boy 3000 from Fallout 4 — monochrome phosphor green on near-black, CRT scan lines, chunky monospace typography, owl mascot artwork.

Before you write any code, please do the following in order:

1. **Read all design files in `docs/`.** These contain the full spec and your build instructions:
   - `docs/GLOSSARY.md` — single source of truth for terms used throughout the spec. Read this first so you have a shared vocabulary.
   - `docs/DESIGN.md` — product spec, visual system, page-by-page requirements
   - `docs/ARCHITECTURE.md` — technical structure, file tree, component contracts
   - `docs/REPO_SETUP.md` — repo structure, distributed CLAUDE.md pattern, LESSONS.md convention, Milestone 0 instructions
   - `docs/TOKEN_DATA_RESEARCH.md` — open questions about Claude Code's log format that you must investigate before writing the parser
   - `mockups/*.html` — static visual reference. Open any of them in a browser to see the Pip-Boy aesthetic, layout patterns, and component styling. Generated by `build_all.py` at the repo root. **Important: the mockups predate several spec revisions.** Use them for *visual* reference (colours, spacing, typography, mascot placement, chart treatment, status bar styling) but treat **DESIGN.md as authoritative** for *content* (which stats appear, which sub-pages exist, which features are in v1 vs v2). Where they disagree, DESIGN.md wins.

2. **Inspect my actual Claude Code logs.** Look in `~/.claude/projects/` and pick a recent session JSONL file. Pretty-print 10–20 lines and tell me what fields you see. Use this to answer the questions in `docs/TOKEN_DATA_RESEARCH.md` before writing the parser. **This is the most important step** — every other piece of code depends on getting the parser right, and the log format is undocumented.

3. **Update TOKEN_DATA_RESEARCH.md** with what you find. Replace the "open question" sections with confirmed answers. If any of my assumptions in DESIGN.md need to change because the data isn't actually available (e.g. context can't be decomposed by source), tell me and update DESIGN.md too.

4. **Confirm the plan with me before writing code.** Once you've inspected the logs and updated the research doc, write a short summary of:
   - What you found in the logs
   - Which DESIGN.md features are buildable as specified
   - Which features need to fall back to estimates or get cut
   - Your proposed build order (which milestones, in what order)
   
   Wait for me to approve before starting on code.

5. **Build in milestones, not all at once.** I'd like to see and test progress at each stage rather than waiting for a big-bang reveal. Suggested milestones:
   - **M0: Repo setup.** Follow `docs/REPO_SETUP.md` exactly. Initialise the repo, create the directory structure, write the root CLAUDE.md plus subdirectory CLAUDE.md skeletons, write LESSONS.md, write the initial 16 ADRs, set up CI, push to GitHub publicly. **Do not write any feature code in M0.** It ends with a clean public repo containing only documentation, scaffolding, and the design package.
   - **M1: Skeleton.** Extension activates, opens an empty webview panel with the menu structure and CRT styling. No data yet. I should be able to install it and click through the pages.
   - **M2: Parser + storage.** SQLite schema, JSONL parser, file watcher. No UI yet beyond M1, but the database starts filling up as I use Claude Code.
   - **M3: LIVE pages.** SESSION, CONTEXT, CACHE pages render real data from the database.
   - **M4: STATS and HISTORY pages.** Charts work, all sub-pages functional.
   - **M5: TIPS, ABOUT (with INFO and GLOSSARY sub-pages), onboarding.** Plus settings persistence and dashboard sync modal.
   - **M6: Status bar (in-window only) and polish.** CRT flicker, blip sound, advisory engine.
   - **M7: Tests, packaging, README screenshots.** Ready to ship.

6. **Use the owl PNGs in the `assets/` folder.** They're already there. Don't generate new artwork.

## Things I want you to do without being asked

- **Be honest about uncertainty.** DESIGN.md has a section on "honest limitations" — read it and respect it. Don't fake projection data when you should be showing the LEARNING state. Don't label estimates as facts.
- **Push back on the design if you see a problem.** I'd rather rework the spec than build something that doesn't work. If a feature can't be built reliably, tell me before you build it badly.
- **Explain things in everyday language as you go.** I'm a solo dev with Python experience and I'm newer to TypeScript and VS Code extensions. When you make a non-obvious technical decision, walk me through what it does and why. Don't just commit and move on.
- **Use Git from M0 onwards.** Initialise the repo in M0 per `docs/REPO_SETUP.md`, commit at each milestone, use Conventional Commits (`feat:`, `fix:`, `docs:`, `refactor:`, `test:`, `chore:`). I like to be able to roll back.
- **Maintain the distributed CLAUDE.md files.** When you adopt a new convention or learn a quirk specific to a subsystem, update the closest CLAUDE.md (root, `src/`, `src/parsing/`, `src/webview/`, `src/data/`, `src/domain/`, or `test/`). Keep the root CLAUDE.md short — subsystem-specific notes go in the subdirectory CLAUDE.md, not root.
- **Maintain LESSONS.md.** Whenever a non-obvious bug takes more than 30 minutes to track down, or you discover a platform/library quirk worth remembering, add an entry in `LESSONS.md` using the format in `docs/REPO_SETUP.md`. Read existing LESSONS.md entries before tackling areas marked as previously problematic.
- **Write ADRs for significant decisions.** If you make a non-trivial architectural call during the build (something that future-you would want to understand later), add an ADR in `docs/decisions/` using the format in `docs/REPO_SETUP.md`. Number sequentially after the initial set in M0.

## Things I want you NOT to do

- Don't add any network calls in v1. The extension is fully local. No telemetry, no update checks against a server, no font CDN. Bundle everything. Opt-in analytics is being considered for v2 but is explicitly out of scope for v1 — adding it would require a new ADR superseding 0004.
- Don't add a frontend framework. The webview is plain HTML/CSS/TS. No React, no Vue, no Tailwind.
- Don't use chart libraries. The charts are inline SVG, hand-rolled to match the aesthetic. The mockups in `mockups/*.html` show how — copy that approach. Do not copy any `@import` or font-CDN references if you happen to see them in the mockup CSS; ADR 0004 forbids network calls in the extension.
- Don't add features that aren't in DESIGN.md. If you think of something cool, mention it as a v2 idea but don't build it. I want a focused v1.
- Don't pretend the parser works on log formats you haven't seen. If my Claude Code version logs differently from what you can find references for, tell me and we'll figure it out.
- Don't hardcode pricing anywhere except `src/domain/pricing.json`. Don't hardcode the peak hour schedule anywhere except `src/domain/peakHourSchedule.ts`. Single sources of truth.
- Don't put detailed subsystem context in the root CLAUDE.md. Use the subdirectory CLAUDE.md files. Root stays under ~100 lines.
